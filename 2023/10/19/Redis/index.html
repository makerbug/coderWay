Redis网络模型

Redis采用I/O多路复用技术
为什么Redis要使用IO多路复用这种技术呢，因为Redis是跑在单线程中的，所有的操作都是按照顺序线性执行的，但是由于读写操作等待用户输入或输出都是阻塞的，所以IO操作在一般情况下往往不能直接返回，这会导致某一文件的IO阻塞，导致整个进程无法对其他客户提供服务，Redis使用IO多路复用技术就是为了解决这个问题

多个客户端连接Redis时，Redis会将客户端socket对应的fd注册进epoll，然后epoll同时监听多个文件描述符。

Redis为什么这么快

  1. 基于内存实现
    内存读写相较于磁盘，省去了磁盘IO的消耗
  2. 高效的数据结构
  3. 合理的数据编码
    Redis支持多种数据类型，每种基本类型，不同的应用场景可能对应多种数据结构
  4. 合理的线程模型

Redis高可用

有三种模式实现高可用，主从模式、sential哨兵模式、cluster集群模式

主从模式


主库读写，从库读，主从同步

主从同步存在全量复制和增量复制两种情况

全量复制


从库首次连接主库时，由于之前没有进行同步过，所以首先进行一次全量数据同步到从库

slave主动和master通信，发送psync命令
参数：

  1. 主库id(runId)：第一次不知道masterId，所以为？
  2. 偏移量offset：slave从哪里开始复制，第一次复制
    -1表示全量复制


master收到后发送fullresync命令，表示这是一次全量复制，参数：master的runId

然后就开始传输rdb文件
要点：

  1. bgsave异步生成rdb文件
  2. rdb传输过程中新的写操作写在缓冲区replication buffer中，rdb传输完成后，再传输replication buffer
  3. replication buffer随着连接的断开而释放
  4. master为每一个slave生成一个replication buffer

    生产RDB会阻塞主线程，整个过程很耗资源，如果采用一主多从的方式，那么势必会增加主库的压力，可以选择一个从，再从从库上分裂出一个从或多个从，来减少从库的压力

增量复制

在主从复制过程中，如果网关抖动了或者中断了，redis会重新进行全量复制么
Redis2.8之后，采用增量复制的机制，来减少数据的传出，达到快速复制的目的。
过程：

  1. 全量复制过程中，master会给slave发送offset，slave接收数据之后，会增加offset来记录接收master的多少数据
  2. 如果网络发生了中断，就会重试和master重新连接，连接之后会重新发送自己的offset给Master，Master会根据Slave发送的偏移量来决定给Slave做全量复制还是增量复制

那么，网络中断这段时间，请求写的操作都记录在哪呢
在主从复制过程中，新的写请求再写入replication buffer的同时，也都会写入到一个repl_backlog_buffer的缓冲区内

repl_backlog_buffer是一个环形缓冲区，会记录Master接收新的写请求数据的偏移量和写命令，这样Slave重新再连接之后，就可以从这里接着发送命令给Slave

增量复制的过程
  1. slave发送psync命令，发送自己的offset和Master的runID
  2. master接收到psync之后，判断是进行增量复制还是全量复制

    如果主从节点的offset差距没有超过环形缓冲区的长度，或者没有发生套圈，也就是不会覆盖之前的数据，那么Master就会告诉Slave，可以增量复制

要点

  1. redis实例的内存大小不要设置的太大，能够缩短rdb文件的生成时间，同时更方便在网络中的传输。
  2. 如果master和slave断开连接的而时间超长，那么repl_backlog_buffer缓冲区内的数据很可能就被覆盖了，就会退化为全量复制，因此repl_backlog_buffer可以设置的大一点
  3. replication buffer:这个缓冲区如果从库接受的很慢，这个缓冲区满了，那么redis就很容易OOM，如果满了，主库会断开连接，删除buffer。

哨兵模式每个节点存储数据是一样的，浪费内存，并且不好在线扩容

cluster集群模式


概述

  1. 官网要求六个节点才能保证高可用，即三主三从
  2. 当master节点挂掉后，Redis cluster会从master对应的slave节点中选举一个新的master完成故障转移
  3. redis cluster默认不支持slave节点读或者写的，slave节点更多是为了master发生故障时备用的节点。

节点通信

使用分布式存储中需要提供维护节点元信息的机制，

节点元信息：节点负责哪些数据、是否出现故障等状态信息。

常见的元信息维护方式为：

  1. 集中式：将元信息集中存储在一个外部组件中，如Zookeeper
  2. P2P

每个节点都有一个专门用于节点通信的端口，就是自己提供服务的端口号+10000

redis cluster如何处理实例和数据的映射关系

哈希槽：哈希槽是一个0-16383之间的整数组成的空间，每个槽可以存放一部分数据,只有master节点有槽，从节点没有

当向Redis Cluster中写入数据时，Redis会使用一致性哈希算法来确定数据应该存储在那个哈希槽中，步骤如下：

  1. 根据数据的键，Redis cluster使用CR16算法计算出一个16位的Hash值
  2. 根据哈希槽索引，Redis Cluster确定存储数据的节点。

哈希槽数据迁移

在Redis cluster中，哈希槽迁移是一个自适应的过程，它会根据集群的状态和负载情况动态进行，实现了数据的无损迁移和负载均衡

  1. 节点握手：
    当新节点加入Redis Cluster或者一个节点离开集群时，集群中的其他节点会与新节点建立握手连接。节点之间通过握手来交换集群拓扑信息和状态
  2. 哈希槽重分配
    一旦握手完成，集群会开始哈希槽的重分配。涉及到将一部分哈希槽从现有的负责节点迁移到新的目标节点。哈希槽的迁移是逐个槽进行的，每次迁移一个槽。
  3. 迁移数据同步
    全量同步：发生在初始迁移或网络断开后重新连接时。
    增量同步：全量同步完成后，源节点继续接受和处理写入操作时
  4. 槽迁移确认
  5. 客户端重定向：在迁移过程中，客户端可能会尝试正在迁移的槽，为了确保数据一致性，Redis Cluster会向客户端发送重定向消息，告诉客户端正确的目标节点地址，使其请求到正确的节点。

高可用性与主备切换

  1. 判断节点宕机
    主观宕机：一个节点认为另一个节点宕机 pfail
    客观宕机：多个节点认为另一个节点宕机 fail
    如果在cluster-node-time内，某个节点一直没有返回pong，那么就会被认为pfail
    如果一个节点认为某个节点pfail了，那么会在gossip ping消息中，ping给其他节点，如果超过半数节点都认为pfail了，那么就会变成fail
  2. 从节点过滤
    对宕机的master node，从其所有的slave node中，选择一个切换成master node
  3. 从节点选举
    每个从节点，根据自己对master复制数据的offset，来设置一个选举时间,offset越大的node，优选进行选举
    所有的master node 给被选举的slave投票，超过半数的投票即可让slave晋升为master
    slave执行主备切换，进行故障转移

Redis管道

Redis的传输层是基于TCP协议，一次操作请求的完成，存在网络传输来回的开销。管道的目的就是降低RTT（Round Trip Time）往返时间对性能的影响
在批量操作的前提下，每一次Redis执行命令，都涉及到读Read和写Write的系统调用，系统会将用户端切换到内核段，上下文切换有一定的系统消耗。
使用管道的话，批量操作只需要一个Read和一个Write。

pipelining 使用场景

  1. 对性能有要求
  2. 需要发送多个指令到服务端
  3. 不需要上个命令的返回结果作为下个命令的输入

要点

  1. 客户端使用pipelining发送命令时，服务器端需要消耗内存来存放响应，因此如果需要发送大量的命令，最好分批发送，如一次发送一万个，读取结果，再循环发送剩下的命令。这种方式，速度上几乎无差异，但是很节省内存。

客户端与Redis的交互过程

  1. 客户端进程调用write（）把消息写入操作系统内核为Socket分配的send buffer中
  2. 操作系统会把send buffer中的内容写入网卡，网卡再通过网关路由把内容发送到服务器端的网卡
  3. 服务端网卡会把接受到的消息**写入操作系统为Socket分配的recv buffer
  4. 服务器进程调用read（）读取消息后进行处理。
  5. 处理完成后调用write（）把返回结果写入到服务器端的send buffer
  6. 服务器操作系统再将Send buffer中的内容写入网卡，然后发送到客户端
  7. 客户端操作系统将网卡内容读到recv buffer中
  8. 客户端进程调用read（）从recv buffer中读取消息并返回

Redis 事务

多数秒杀、抽奖、抢红包等大并发高流量的功能一般都是基于Redis实现
Redis事务包含两种模式：事务模式和Lua脚本
Redis事务：

  1. 保证了隔离性
  2. 无法保证持久性
  3. 具备一定的原子性，但无法保证回滚

事务包含三个阶段：

  1. 事务开启，使用MULTI，该命令表示执行该命令的客户端从非事务状态切换至事务状态
  2. 命令入队，MULTI开启事务之后，客户端的命令并不会被立即执行，而是放入一个事务队列
  3. 执行事务或丢弃，如果收到EXEC命令，事务队列里的命令将会被执行，如果是Discard则事务被丢弃。

事务的四大特性

  1. 原子性：原子性是指**事务是一个不可分割的工作单位，其中的操作要么都做，要么都不做

    Redis命令入队时报错，会放弃事务执行，保证原子性
    命令入队时正常，执行EXEC命令后报错，不保证原子性。

  2. 持久性：事务一旦提交，它对数据库的改变就是永久性的
  3. 隔离性：事务内部的操作与其他事务是隔离的，并发执行的各个事物之间不能互相干扰。

    EXEC执行之前，Redis key可以被修改，可以使用WATCH机制来实现乐观锁的效果
    EXEC命令执行后，Redis会保证命令队列中所有命令执行完，可以保证隔离性

  4. 一致性：事务执行前后，数据的改变都是合法的，且各方数据都是一致的。

Lua脚本

背景；
有时候我们需要一次性操作多个Redis命令，但是多个操作不具备原子性，而且Redis的事务也不够强大，不支持事务的回滚，还无法实现命令之间的关系计算，所以，在一般的开发中，我们会利用Lua脚本来实现Redis的事务。
好处：

  1. 减少网络开销，多个请求通过脚本的形式一次发送，减少网络时延
  2. 原子操作，Redis会将整个脚本作为一个整体执行，中间不会被其他命令插入。
  3. 复用，客户端发送的脚本会永久存在Redis中，其他客户端可以复用这一脚本而不需要使用代码完成相同的逻辑。

加锁流程

流程如下：

  1. A、B争抢一把锁，A获取到后，B阻塞
  2. B线程阻塞时**并非主动CAS，而是PubSub方式订阅该锁的广播消息
  3. A操作完成释放了锁，B线程收到订阅消息通知
  4. B唤醒开始继续抢锁，拿到锁。
